# Clarify Requirement

透過結構化訪談幫助工程師釐清開發需求，產出一份 `docs/ideas/{slug}.md` 作為後續 `write-a-prd` 的輸入。

> **與 `grill-me` 的差異**：`grill-me` 是通用設計壓測，適用任何計畫或架構討論。`clarify-requirement` 專注於需求釐清，有固定的輸出文件格式，且下游明確接 `write-a-prd`。

---

## 執行步驟

### 第一步：逐回合探索

使用者確認無誤後，開始訪談，從各種理解角度出發，把需求挖深、挖清楚。每回合只問**一個目前最關鍵的問題**，以 **A/B/C 多選一**格式呈現，降低使用者思考成本。

#### 提問格式（每回合）

```
---
問題 N：[問題標題]

[一到兩句情境說明，讓使用者知道為何要問這題]

 - A）[選項一]
 - B）[選項二]
 - C）[選項三，視情況加]

我的建議：B（[選項關鍵字]），理由是：
 - 1）[理由一]
 - 2）[理由二]
 - 3）...

你們的情況是哪一種？
```

選項設計原則：
- 涵蓋最常見的 2–4 種情況，選項之間互斥
- 若情況較複雜，可加「D）以上都有，但…」之類的複合選項
- 「我的建議」要有明確理由，讓使用者快速判斷是否同意，而非只是複述選項

#### 決策樹原則

訪談的目標不是把問題清單逐一打勾，而是把整棵決策樹走到底。每次使用者回答後，主動做兩件事：

1. **識別衍生分支**：這個回答是否開了新的子問題？若有，立刻接著問，把這條分支走到底，再回到下一個主題。不要廣度優先地跳來跳去——深度優先，一條路走完再換一條。

2. **明確標示依賴**：若後續問題的答案取決於剛才的決定，在提問時說出來。例如：「因為你選了 B（多租戶），接下來我需要確認資料隔離的策略…」這讓使用者看清楚決策之間的邏輯關係，也幫助你自己不跳過關鍵分支。

**目標是毫不放水（relentless）**：不斷問下去，直到每條分支都有明確答案或被標記為「未解決問題」，才停止。

#### 處理「我不知道」

若使用者回答不知道，不要直接留空，先試著幫他想：
1. 提供一個具體情境範例，讓他用直覺判斷（例如：「假設有兩個客戶 A 和 B，他們的資料要分開嗎？」）
2. 若仍無法判斷，從反面切入（例如：「那你確定哪些情況**不需要**隔離？」）
3. 若兩種方式都無效，才將這個問題記錄到「未解決問題」，繼續下一個分支

#### 探索 Codebase

如果問題可以透過探索 Codebase 解決，使用另一個 Agent 探索，避免污染訪談 Context。

### 第二步：收斂追蹤與補問

每回合提問後，在內心評估三個核心錨點的填充品質。這三個錨點是**最終文件的必要骨架**，其他欄位由訪談自然產出：

| 區塊 | 收斂標準 |
|------|---------|
| 背景與動機 | 清楚說明為何要做、解決什麼問題或創造什麼價值 |
| 目標 | 有具體的成功定義，不是空泛描述 |
| 未解決問題 | 列出需要確認的問題，或使用者確認「目前沒有未解問題」 |

若探索問完後某個區塊仍然空白或薄弱，針對該區塊補問一題，再做一次評估。

訪談過程中產出的其他內容（使用者角色與情境、資料範疇、Out of Scope 等）照實記錄進文件，不受格式約束。

當所有區塊品質達標，主動提示：
> 「訪談已接近收斂，目前還有 X 個未解問題待確認。要繼續深挖，還是這樣就可以產出摘要？」

### 第三步：存檔

使用者確認收斂後，詢問：「是否存檔？」

**若選擇存檔：**

1. 根據需求描述自動生成 slug（英文、小寫、用 `-` 連接，例如 `user-notification-system`）
2. 執行：
   ```bash
   cp <skill-path>/references/clarify-template.md docs/ideas/{slug}.md
   ```
   > 若 `docs/ideas/` 不存在，先執行 `mkdir -p docs/ideas`
3. 將訪談內容以**繁體中文**填入對應區塊
4. 提示使用者將 `docs/ideas/` 加入 `.gitignore`（若尚未加入）
5. 提示下一步：
   > **接下來你可以：**
   > - **開新 session**：開新對話，執行 `/write-a-prd docs/ideas/{slug}.md`
   > - **同 session 繼續**：直接在此執行 `/write-a-prd docs/ideas/{slug}.md`

**若不存檔：**
直接提示上述兩個選項，由使用者決定是否手動儲存。

## 訪談規則速查

| 規則 | 說明 |
|------|------|
| 每回合一題 | 不要一次丟多個問題，讓使用者逐一聚焦 |
| A/B/C 選項格式 | 每個問題用選項列出 2–4 種情況，並附「我的建議：X）—— 理由」，讓使用者選字母即可回答 |
| 繁體中文輸出 | 所有訪談內容與文件一律用繁體中文 |
| 不探 codebase | 預設不主動讀程式碼，除非使用者明確要求，探索時需使用另一個 Agent 處理，避免污染 context |
| 不寫 PRD | 需求釐清完成後交給 `write-a-prd` |
| 不寫 User Stories | User Stories 留給 `write-a-prd`；訪談只問「誰要用、在什麼情境、為了什麼」，不整理成 AS a/I want/so that 格式 |
| 不寫 AC | Acceptance Criteria 留給 `write-a-prd` |
| 不拆 task | task 拆解留給 `prd-to-issues` |
| 不寫 code | 這個 skill 只做訪談與文件 |
