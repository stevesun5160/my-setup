# SKILL.md 品質評審標準

供自我審查與手動審查使用。

## Frontmatter

| 項目 | 規則 |
|------|------|
| name | 1–64 字元、小寫字母 + 數字 + 單個 hyphen、不得連續 hyphen、與目錄名完全一致 |
| description 長度 | ≤ 250 Unicode 字元（軟性上限）；平台硬限 1,024 字元 |
| description 語態 | 第三人稱陳述 skill 功能 |
| description 觸發詞 | 含正向觸發情境 + 明確負面觸發（「不適用於⋯」）|

## 結構

| 項目 | 規則 |
|------|------|
| SKILL.md 行數 | < 500 行；超過時警示並建議搬移至 `references/` |
| 子目錄深度 | 最多一層（`references/auth.md`，不可 `references/db/v1/auth.md`）|
| 禁止建立文件檔 | 不建立 `README.md`、`CHANGELOG.md` 等說明型檔案 |
| 必要段落 | 概覽 / 使用時機 / 核心流程 / 常見合理化藉口 |
| 選用段落 | 警訊 / 錯誤處理 / 驗證 / 延伸參考（依複雜度納入）|

## 內容品質

| 項目 | 規則 |
|------|------|
| 語態 | 第三人稱祈使句（「擷取⋯」而非「你應該擷取⋯」或「我會擷取⋯」）|
| 步驟 | 嚴格時序編號，有決策分支時明確列出（「若⋯執行 X，否則跳至步驟 N」）|
| 術語一致 | 同一概念全文用同一個詞，不混用（例：不可前段用 `refactor`、後段用「重構」）|
| 精簡 | 無裝飾性廢話、無重複贅述、無 agent 本來就能處理的多餘指示 |
| 邏輯一致 | 前後敘述一致、條件分支不衝突、流程銜接合理 |
| 英文大小寫 | 依慣例：`SKILL.md`、`frontmatter`、`Markdown`、`JSON`、`GitHub` |
| 中文標點 | 正文避免破折號（`——`）與分號（`;`），以逗號或句號替代 |

## Progressive Disclosure

| 項目 | 規則 |
|------|------|
| JIT 載入 | 明確指示 agent 何時讀子目錄檔案（「讀取 `references/auth.md` 以⋯」）|
| 路徑格式 | 使用相對路徑 + 正斜線（`references/schema.md`，非絕對路徑）|
| 大型 reference | > 300 行時附目錄 |

## 子目錄選擇準則

| 類型 | 放置位置 |
|------|---------|
| 易錯或重複性工作（解析、查詢、格式轉換）| `scripts/` |
| 大型 schema、API 速查表、領域知識 | `references/` |
| 固定輸出格式、模板、靜態檔案 | `assets/` |

Scripts 只放單一用途的小型腳本。長期函式庫程式碼屬於標準 repo 目錄，不放在 `scripts/`。

## 設計合理性

| 項目 | 規則 |
|------|------|
| T1 觸發條件特異性 | 正向觸發是否精確可辨識、負向觸發是否能排除常見誤觸情況 |
| T2 流程完整性 | 核心流程是否覆蓋 description 所有使用情境，無遺漏分支 |
| T3 範圍適切性 | skill 是否做太多事（過寬）或只處理單一邊緣情境（過窄）|
| T4 錯誤條件處理 | 邊界條件與失敗情境是否有明確處理指引 |
