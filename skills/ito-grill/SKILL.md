# ito-grill

## 概覽

逐一追問計畫或設計的每個決策分支，直到達成共識。訪談收斂後，視使用者需求將 summary 存至 `docs/idea/` 或直接銜接下一個 ito pipeline。

## 使用時機

- 使用者說「我想討論⋯」或「幫我釐清⋯」
- 需求模糊，不確定要走哪條 pipeline
- 需要壓力測試一個計畫或設計

**不應使用的情況：** 需要直接實作、需求已明確且使用者知道要走哪條 pipeline。

## 核心流程

### 步驟 1：逐一追問

逐一追問計畫或設計的每個決策分支，直到與使用者達成共識。依照決策樹的每個分支逐步深挖，先解決有依賴關係的決策。

**四條守則：**
1. 一回合只問一題。
2. 每當前一個決策解鎖新子問題，且有明確因果關係時，先用一句話宣告依賴關係，再提下一題。
3. 若問題可透過探索 codebase 回答，優先探索，不直接問使用者：
   - 單一明確查詢（e.g., 找一個 function 名稱）→ 直接用 Grep / Glob
   - 範圍不確定或需要多輪搜尋 → 使用 sub-agent（Explore 類型），避免污染主對話 context
4. 根據問題性質選擇對應格式（見「問題格式」章節）。

### 步驟 2：收斂判斷

當所有關鍵分支已達成共識，主動提示，並列出所有未解問題供使用者選擇，最後一個選項固定為「進入下一步」：

> 訪談已接近收斂，目前還有 X 個未解問題待確認：
> - A）[未解問題 1]
> - B）[未解問題 2]
> - C）進入下一步

使用者也可在任何時候主動說「結束」或「可以了」觸發收斂。

### 步驟 3：收斂後動作

詢問使用者：

> 要將 summary 存到 `docs/idea/` 供之後參考，還是直接銜接下一個 skill？

**若選擇存檔：**
1. 根據訪談主題自動命名（e.g., `docs/idea/user-auth-refactor.md`）。
2. Summary 格式根據訪談內容自判最佳呈現方式，不綁定固定結構。
3. 建立檔案，確認路徑後告知使用者。

### 步驟 4：Pipeline 推薦

無論是否存檔，根據訪談內容推斷最適合的下一個 ito pipeline，提供推薦與理由：

| Pipeline | 適用情境 |
|----------|---------|
| `/ito-prd` | 有明確功能需求待展開 |
| `/ito-debug` | 要修 bug，根因尚待確認 |
| `/ito-refactor` | 要重構，需先評估影響範圍 |
| `/ito-spike` | 技術方向不確定，需要先調研 |

## 問題格式

根據問題性質挑選格式：若能列出 2–4 個互斥選項，優先採用決策型或現況確認型。只有當問題本質發散時才使用開放型。

### 格式 1 — 決策型

使用者需要在幾個方向中做出設計或產品決策，且不同選項有不同的技術含義。**必須附推薦與理由。**

```
---
問題 N：[問題標題]

[一到兩句情境說明，讓使用者知道為何要問這題]

 - A）[選項一]
 - B）[選項二]
 - C）[選項三，視情況加]

我的建議：B（[選項關鍵字]），理由是：
 - 1）[理由一]
 - 2）[理由二]

你傾向哪個方向？
```

### 格式 2 — 現況確認型

了解現有系統、流程或情境；使用者自己最清楚，**不需要附推薦**。

```
---
問題 N：[問題標題]

[一到兩句說明為何需要確認這個現況]

 - A）[可能情況一]
 - B）[可能情況二]
 - C）[其他情況]

你們目前是哪一種？
```

### 格式 3 — 開放型

問題本質發散（背景脈絡、使用者樣貌、動機來源），列選項反而會強行收斂。

```
---
問題 N：[問題標題]

[一到兩句情境說明，點出為何這題需要開放回答]

你的情況是？
```

**選項設計原則：**
- 涵蓋 2–4 個最常見的情況，選項之間互斥。
- 若情境複雜，可加複合選項：`D）以上都有，但⋯`。
- 開放題拿到答案後，立即以決策型或現況確認型往下深挖。

## 常見合理化藉口

| 合理化藉口 | 實際情況 |
|-----------|---------|
| 「這題很簡單，不需要問」 | 跳過追問會累積隱性假設，導致後續 pipeline 方向錯誤 |
| 「使用者說可以了，不用再問」 | 收斂判斷由 agent 主導，確認所有關鍵分支都已覆蓋再收斂 |
| 「一次問多題比較快」 | 多題並問會讓使用者遺漏分支，破壞決策樹的完整性 |

## 警訊

- 一回合出現兩個以上問題
- 問題沒有選項，也沒有開放情境說明
- 決策型問題缺少推薦與理由
- 跳過 codebase 探索直接問使用者技術細節
- 使用者回答超出選項範圍，agent 直接接受後進下一題，未追問

## 驗證

- [ ] 所有關鍵決策分支已覆蓋，無跳過
- [ ] 每題使用正確格式（決策型 / 現況確認型 / 開放型）
- [ ] 收斂提示已發出，使用者確認進入下一步
- [ ] 若選擇存檔，`docs/idea/[主題].md` 已建立

## 錯誤處理

- 若使用者在訪談中途說「先跳過這題」，記錄為未解問題，計入收斂提示的 X 值。
- 若使用者回答「我不知道」：先用反向問題追問（e.g., 從結果、影響或相反情境切入），嘗試引導出答案；若依然是不知道，記錄為未解問題，跳過繼續訪談，計入收斂提示的 X 值。
- 若使用者的回答超出選項範圍（e.g., 描述了一個未列出的情況），不直接接受後進下一題，而是根據該回答追問，釐清其技術含義或決策影響，直到充分理解再繼續。
- 若推薦 pipeline 後使用者選擇其他 pipeline，尊重使用者選擇，帶著 summary context 銜接。
