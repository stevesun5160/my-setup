# ito-prd

## 概覽

透過逐題深度追問，將使用者需求收斂為可執行的 PRD，以固定模板輸出後存至本地或 gh issue。

## 使用時機

- 使用者說「寫 PRD」、「整理需求」、「把這個 feature 寫成文件」
- 使用者說「開需求 issue」、「把這個 feature 開成 issue」
- 使用者提及要更新或修改某個 issue 編號或本地 PRD 路徑
- 需求討論或腦力激盪收斂後，使用者表示想整理為 PRD 文件

**不應使用的情況：** 需要直接實作功能、純技術架構設計討論、非需求文件的一般筆記、需要拆分 sub-task 而非撰寫需求文件。

## 核心流程

### 步驟 1：判斷意圖

從 prompt 判斷使用者意圖：

- **修改既有 PRD**：使用者提及要更新、修改某個 issue 編號或 `.md` 路徑 → 記錄來源，進入步驟 2 讀取既有內容（最終將建立新 PRD，完成後詢問是否關閉舊的）
- **建立新 PRD**：其他情況 → 跳過步驟 2，直接進入步驟 3

若意圖不明確，追問一次，仍不明確則預設建立新 PRD 並告知使用者。

### 步驟 2：讀取既有 PRD

- 本地路徑：讀取指定 `.md` 檔案
- issue 編號：以 `gh issue view {編號}` 取得 body，記錄 issue 編號供步驟 6 使用

讀取後作為步驟 3 訪談的補充背景，優先追問尚未覆蓋的欄位。

### 步驟 3：逐一訪談

訪談使用以下三種問題格式：

- **決策型**（使用者需選方向）：說明情境、列出 2–4 個互斥選項，**附上推薦與理由**
- **現況確認型**（了解既有狀態）：說明原因、列出可能情況，不附推薦
- **開放型**（本質發散的背景問題）：說明為何需要開放回答，直接問，**附上預期或假設**

若能列出互斥選項，優先用決策型或現況確認型。只有問題本質發散時才用開放型。取得開放回答後，立即以決策型或現況確認型往下深挖。

**核心守則：**
1. 一回合只問一題，追問留至下一回合
2. 每個回答都是下一個問題的起點，找出其隱含假設或潛在弱點再繼續，不因回答聽起來合理就放過
3. 前一決策解鎖新子問題時，先一句話說明依賴關係，再提問
4. 技術問題優先探索 codebase，取得結果後一句話複述並請使用者確認
5. 以簡潔且通順的文字來提問和建議，講重點，不堆砌專有名詞或冗長論述
6. 若單題回答不足以填入完整欄位，繼續追問不同角度、例子或反面情境，直到該欄位能寫出具體內容為止。

### 步驟 4：收斂判斷

確認以下所有欄位均有足夠深度的答案，且核心決策分支已達共識。每個欄位的答案須達到能寫出具體內容的程度，初步回答不算收斂。

| 欄位 | 訪談目標 |
|------|---------|
| 現況 | 使用者目前在什麼情境下做什麼事 |
| 痛點 | 現行做法有什麼問題、不足或摩擦 |
| 目標 | 功能成功後會達成什麼結果 |
| US 角色 | 每個 US 的主體是誰 |
| US 目標 | 該角色想達成什麼 |
| US 效益 | 達成後獲得什麼價值 |
| AC 主路徑 | 每個 US 至少一個主要驗收情境 |
| AC 邊界情境 | 可辨識的邊界情境（不存在時可跳過，但應主動探問） |
| Out of Scope | 這次刻意不做的功能或情境 |
| 已知侷限 | 硬限制（rate limit、權限、第三方依賴）與已知風險 |

User Stories 盡可能多列。當使用者描述不同角色、情境或邊界情境時，主動拆成新 US。簡單 feature 至少 2–3 個 US，若能辨識多個角色或情境，通常 5 個以上。

**若有未解問題**，列出後詢問：

> 核心決策已達共識，剩餘 X 個未解問題待確認，要繼續討論哪個問題，還是收斂？
> - A）[未解問題 1]
> - B）[未解問題 2]
> - C）收斂，進入下一步

**若無未解問題**，直接宣告收斂後進入步驟 5，不發送選項提示。

使用者隨時說「結束」或「可以了」時，agent 先列出尚未覆蓋的欄位（若有），告知使用者後詢問是否仍要收斂，若無缺口則直接進入步驟 5。

### 步驟 5：產出 PRD

讀取 `assets/prd-template.md` 依訪談內容填入：

- **問題描述**：現況、痛點、目標
- **User Stories**：每個不同角色、情境、邊界情境各自獨立成一個 US，盡可能窮舉，寧多勿少（簡單 feature 至少 2–3 個，能辨識多個角色或情境時通常 5 個以上）。每個 US 含角色/目標/效益，加 AC 表格
- **AC 表格**：每個 US 至少一列主路徑，能辨識時補上邊界情境
- **Out of Scope**：訪談中「這次不做」的項目
- **已知侷限**：硬限制與已知風險
- **開放性問題**：僅當存在未解問題時才加入，否則整段省略

### 步驟 6：選擇輸出方式

詢問使用者：

> PRD 草稿已完成，要存到本地還是建立 gh issue？
> - A）存本地（預設 `docs/ito-temp/prd/`，可自訂路徑）
> - B）建立 gh issue（label: `PRD`）

**本地分支：**
- 依訪談主題自動命名（例 `user-auth-refactor.md`），建立新檔
- 若非預設路徑，執行前確認
- 完成後回報實際寫入路徑

**gh issue 分支：**
- title 格式：`[PRD-{issue 編號}] {PRD 名稱}`
- body：完整 PRD markdown（移除第一行 H1 標題，避免與 issue title 重複）
- label：固定帶 `PRD`（顏色 `#0075ca`），若 repo 無此 label 先建立
- 初始建立時 title 僅填 `{PRD 名稱}`，取得 issue 編號後更新 title 補上前綴
- 完成後回報 issue URL
- **若步驟 2 記錄了舊的 issue 編號**，建立完成後詢問：「要順手關閉舊的 issue #{舊編號} 嗎？」若使用者確認，執行 `gh issue close {舊編號}`，並在舊 issue 留 comment：「已由 #{新編號} 取代」

## 常見合理化藉口

| 合理化藉口 | 實際情況 |
|-----------|---------|
| 「這題很簡單，不需要問」 | 跳過追問會累積隱性假設，產出空洞的 PRD |
| 「使用者說可以了，不用再問」 | agent 先列出尚未覆蓋的欄位告知使用者，再由使用者決定是否收斂 |
| 「一次問多題比較快」 | 多題並問讓使用者遺漏分支，破壞決策樹完整性 |
| 「這個欄位問一題就夠了」 | 單題通常無法涵蓋完整欄位，薄弱回答必須繼續追問 |
| 「US 只要 1 個就夠了」 | US 數量直接影響後續任務拆分與測試覆蓋 |
| 「開放性問題空著寫『無』比較工整」 | 無未解問題時應省略整個段落，不補「無」 |
| 「技術細節也順便問一問」 | 訪談聚焦產品層面，深入技術會稀釋需求討論 |
| 「建完新 issue 就好，舊的讓使用者自己處理」 | 建完後務必詢問是否關閉舊 issue，不讓使用者自行善後 |

## 警訊

- 一回合出現兩個以上問題
- 決策型問題缺少推薦與理由
- 使用者回答超出選項範圍，未追問即繼續
- 跳過 codebase 探索直接問已可查證的技術事實
- 使用者回答聽起來合理就放過，未探問假設或邊界條件
- 產出 PRD 缺少固定段落
- gh issue 更新後的 title 仍缺少 `[PRD-{編號}]` 前綴，或編號與實際 issue 編號不符
- 無未解問題卻仍產出開放性問題段落（包含只留標題、不填內容）
- 使用者提及既有 issue 或 PRD 路徑，建立新 issue 後未詢問是否關閉舊的

## 驗證

- [ ] 所有 PRD 欄位均已覆蓋（見步驟 5 欄位表）
- [ ] 所有欄位答案達到能寫出具體內容的程度，非僅初步回答
- [ ] 每題使用正確格式（決策型／現況確認型／開放型）
- [ ] 收斂提示已發出，使用者確認進入下一步
- [ ] PRD 結構與 `assets/prd-template.md` 一致
- [ ] 無未解問題時，開放性問題段落已省略
- [ ] 本地輸出：檔案已寫入並回報路徑
- [ ] gh 輸出：issue 已建立，帶 `PRD` label，title 格式正確，已回報 URL
- [ ] 修改既有 PRD 分支：建立新 issue 後已詢問是否關閉舊的

## 錯誤處理

- 若使用者說「先跳過這題」，記錄為未解問題，計入收斂提示，產出時列入開放性問題。
- 若使用者回答「我不知道」，先用反向追問（從結果、影響或相反情境切入）引導，仍無答案則記錄為未解問題。
- 若使用者回答超出選項範圍，追問釐清其產品含義，充分理解後再繼續。
- 若步驟 2 指定的檔案或 issue 不存在，告知使用者並詢問是否繼續建立新 PRD。
- 若 gh 分支建立 `PRD` label 失敗，告知使用者並暫停等待指示。
